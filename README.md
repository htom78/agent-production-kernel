# Agent Production Kernel

Agent Production Kernel is a small, runnable reference implementation of the
four-phase system sketched from OpenMontage:

1. **Kernel:** pipeline manifests, artifact schemas, tool registry,
   checkpoints, decision logs, review, replay hooks, and a `RunEngine` that
   enforces stage order and artifact lineage.
2. **Domain Packs:** software engineering, research, and design-production
   packs, all loaded through `packs/registry.json`.
3. **External Harness/Loop Adapters:** contracts for SWE-bench/SWE-agent style
   patch loops, OpenHands-style sandbox runtimes, LangGraph-style durable
   execution, and eval/trace systems.
4. **Multi-Agent Layer:** explicit roles, handoff artifacts, and audit rules
   that sit on top of the kernel rather than replacing it.

The point is not to make agents "more clever in the moment." The point is to
turn agent work into an inspectable production system: every failure creates
structured evidence, every fix strengthens a system constraint, and every claim
has a replayable verification path.

## Quick Verify

```bash
python3 scripts/verify.py
python3 scripts/verify_real_repo_corpus.py
python3 -m unittest discover -s tests
python3 scripts/replay_regressions.py
python3 scripts/self_assess.py
python3 scripts/battle_report.py
```

Core kernel checks use the Python standard library. The design pack's prototype
render gate also requires the local `playwright` command and Python Playwright
runtime declared in `packs/design/tool_registry.json`.

## Evidence Boundary

This public repository is source-only. Local runtime evidence under `.apk/`,
trace files, cloned public corpus checkouts, and cache directories are ignored.
Replay fixtures in `examples/` are the portable evidence surface.

Treat multi-agent battle reports as fresh release evidence only when the
`agent_battle_harness_report` is bound to the exact `self_assessment_report`
and `battle_report` pair from the same evaluation cycle. Older battle artifacts
remain useful audit history, but they should not silently promote a newer run.

## Core Rule

The kernel owns the control plane. External frameworks may provide execution,
evaluation, traces, or sandboxes, but they do not become the source of truth.
The source of truth is always:

```text
pipeline manifest + artifact schemas + checkpoints + decision log + verifier
```

The verifier discovers domain packs from `packs/registry.json`, runs executable
scenarios for registered pack pipelines, and rejects missing skill files,
unknown tools/artifacts, bad stage ordering, unrelated checkpoint artifacts,
and mismatched decision-log run IDs.
Role policies separate `review_by` from `approval_by`: approval-sensitive
stages must carry an explicit `approval_decision` artifact from the configured
approver role, and that artifact must reference a completed
`approval-decision` source checkpoint owned by the approver. Machine approvers
can be generated by the runtime approval gate; `human_operator` cannot. Human
approval requires an explicit `human_approval_grants` entry, otherwise the stage
is checkpointed as `awaiting_human`. Malformed grant containers fail as contract
errors instead of being treated as a normal human wait state.

## Self-Driving Loop

The project should use itself to continue itself. `scripts/self_assess.py` runs
the verification gates, emits a schema-validated `self_assessment_report`, scores
the current kernel across contract integrity, execution reality, evidence trust,
replay strength, multi-agent readiness, and autonomy, then names the next
highest-priority actions. See `docs/SELF_DRIVING.md`.

Replay scenarios now check artifact fields, checkpoint metadata/review fields,
and negative mutation cases. Battle reports preserve architect, test engineer,
code reviewer, and critic disagreement as a schema-validated artifact rather
than leaving it only in chat history.
Blocking work appears in `battle_report.next_actions`; ongoing release hygiene
such as maintaining the verified corpus and rerunning gates appears in
`battle_report.release_disciplines`, so a clean system does not keep creating
the same self-driving action. Agent Battle readiness checks that field and
carries it into the harness outcome.

`scripts/agent_battle_harness.py` adds the Agent Battle validation layer for
the kernel. It records per-judge context hashes, evidence source labels,
blind-review flags, cross-examination, critic veto state, judge audit checks,
and the final outcome as `agent_battle_harness_report`. Its job is to make
evaluation of the kernel auditable without overstating the evidence: a local
report derived from `self_assess` and `battle_report` is labeled
`evidence_mode=derived_report` and must hold, while an advance requires explicit
external judge reports labeled `external_agent_report`, bound to the same run
ID, carrying advancing verdicts and scores at or above the minimum, and
containing exactly one report for each required role from distinct
`codex-subagent://` sources. Self-assessment only accepts an independent battle
artifact when the harness and semantic validator can read real input report
files, bind `self_assessment_run_id` and `battle_report_run_id` to those files'
actual `run_id` values, show that all non-evaluation self-assessment dimensions
meet the pre-battle readiness gate, show that the bound reports have no
unresolved non-battle next action, show the battle artifact is newer than
the current source evidence, and reject reuse when a newer unresolved
self/battle report exists for the same `evidence_fingerprint`. This avoids the
circular requirement that an
independent battle must already be advanced before it can advance. The kernel's
product goal remains the production control plane rather than a multi-model
battle platform.
Independent hold or veto reports are preserved as real evidence, but they move
the next action to repairing the recorded findings instead of being counted as
advance readiness.

Replay also covers non-happy-path checkpoint branches. The
`checkpoint_branch_replay` fixture preserves `awaiting_human`, `failed`, and
`blocked` states so recovery and approval boundaries can be tested without
pretending every run completed.

`scripts/run_next_action.py` is the bounded autonomy runner. It executes only
allowlisted local verification commands. Unknown handlers, external repos,
credentials, network-cost boundaries, destructive operations, and user-choice
boundaries stop as `blocked` checkpoints in the `kernel-autonomy` pipeline.
When self-assessment has no remaining next action, the runner records an
explicit `no_action` autonomy report with an empty action payload and a
completed checkpoint instead of leaving the stop state implicit.

Scores are intentionally calibrated as readiness signals, not completion
claims. The self-assessor now caps confidence until pack-general execution,
semantic false-green tests, and a broader real-repo corpus exist.

The real-repo corpus target is explicit in
`examples/real_repo_corpus_manifest.json`: at least five non-author repositories
and three failure families. The current corpus includes five BugsInPy-backed
public third-party repos (`youtube-dl`, `tornado`, `ansible`, `thefuck`, and
`black`) plus the controlled dogfood repo, so
`scripts/verify_real_repo_corpus.py` reports `target_met=true`. The generated
`real_repo_corpus_report` also carries an `approval_boundary`: while
`target_met=false`, the status must be `awaiting_human`; once the target is met,
the status must be `not_required`. It also carries `freshness`, which separates
current producer-roundtrip validation of checked-in artifacts from a recent
live external rerun. Live `.apk/real-bug-runs/<run_id>` files only count as
`live_external_rerun` when every manifest entry has one and both command
timestamps are within the configured freshness window; stale live files remain
auditable but report `external_execution=false`. Each corpus entry records its
failing and passing command timestamps, and semantic validation recomputes the
report-level oldest/latest timestamps from those entries. Self-assessment may
pass on the checked-in corpus, but `evidence_trust` is capped below perfect
while live external rerun artifacts are stale.

## Real Repo Bug Proof

The kernel has one public dogfood bug run:

```bash
python3 scripts/run_real_bug_scenario.py
```

It uses `https://github.com/htom78/apk-real-bug-demo`, proves the bug on `main`,
proves the fix on `fix/privacy-cache-key`, writes kernel checkpoints under
`.apk/real-bug-runs/real-repo-privacy-cache/`, and preserves the replay fixture
as `examples/real_repo_bug_run_fixture.json`.

The broader public corpus is generated with:

```bash
python3 scripts/run_bugsinpy_corpus.py --write-examples
```

This runner uses public BugsInPy metadata to check out buggy and fixed commits,
copies the fixed regression test into the buggy checkout, runs the focused
test in an isolated local venv, records failing and passing command evidence,
and writes `real_repo_bug_run` artifacts for the corpus manifest. It does not
use credentials, paid services, pushes, branches, or pull requests.

## Design Pack

`packs/design/` adapts ideas from the MIT-licensed
[`Trystan-SA/claude-design-system-prompt`](https://github.com/Trystan-SA/claude-design-system-prompt)
repo into APK's contract layer. The upstream project is a prompt and skill
library; APK treats it as domain knowledge, not as the control plane.
The pack registers its own semantic validators through `packs/registry.json`,
so design-specific rules do not live in the kernel core. The prototype stage
uses a Playwright-backed browser probe instead of treating a readable HTML file
as render evidence.

The `design-review` pipeline turns that knowledge into structured artifacts:

```text
design_brief -> design_context -> design_prototype_report
  -> accessibility_audit -> visual_quality_report -> design_release_report
```

This makes design work auditable: accessibility blockers cannot be marked pass,
open AI-template or interaction-state findings cannot become a passing visual
quality report, and a design release is ready only when its evidence gates pass.
The demo scenario also preserves local source summaries and a real HTML
prototype fixture so render and provenance claims are file-bound.
