# Architecture

Agent Production Kernel is a control-plane contract for agentic software work.
It borrows the useful shape of OpenMontage and translates it from video
production to general software production.

## Control Plane

The agent is still the active planner and implementer, but it is constrained by
contracts:

```text
PipelineManifest -> Stage Skill -> Tool Registry -> Artifact -> Review -> Checkpoint
```

No stage is considered complete because an agent says it is complete. A stage is
complete when its artifact validates, review passes, required input artifacts
are present in prior checkpoints, and `RunEngine` writes a checkpoint in
manifest order.

## Self-Driving Layer

The kernel also carries a dogfood loop. `scripts/self_assess.py` runs the current
verification gates, produces a `self_assessment_report`, scores the system, and
selects the next actions. This keeps future progress inside the same artifact,
evidence, and replay discipline as the work the kernel expects from other
agents.

`scripts/battle_report.py` turns multi-view disagreement into a structured
`battle_report`, so architecture, test, code-review, and critic judgments can be
preserved as evidence rather than disappearing into chat history.

## Domain Packs

A domain pack provides:

- pipeline manifests,
- artifact schemas,
- tool registry entries,
- stage skills,
- optional roles,
- optional external adapters.

Packs are discovered through `packs/registry.json`. The software pack is the
first domain pack; the research pack is the first non-software proof that the
core control plane can run another domain through manifest, roles, executor,
checkpoint, and replay. Packs can also register artifact semantic validators
through an optional `semantic_module` entry, so domain-specific policy remains
owned by the pack rather than hardcoded into the kernel core. The design pack
then tests a higher-judgment domain: it turns a prompt/skill library into
inspectable design artifacts, accessibility gates, visual quality gates, and
release evidence.

## Why External Tools Are Adapters

Frameworks such as patch-loop benchmarks, sandboxes, durable graph runtimes, and
LLM eval tools are valuable, but they solve different parts of the system. The
kernel deliberately keeps them outside the source of truth.

This avoids a common failure mode: mistaking a green framework run for a complete
product or process fix.

## Evidence Model

Verification evidence is structured. A command result records command text,
status, exit code, stdout/stderr digests, commit SHA, tool version, timestamp,
and artifact references. Free-text evidence is still allowed in domain reports,
but release/verification gates use command evidence.

Golden replay scenarios assert both artifact content and checkpoint metadata,
including actor roles and review decisions. Checkpoints are namespaced by
`run_id/pipeline`, so separate pipelines can safely reuse a run ID without
overwriting stages with the same name. Role policies separate review from
approval. `review_by` names roles that can perform schema/semantic review,
while `approval_by` names roles that must be represented by an explicit
`approval_decision` artifact before the checkpoint can complete. Approval is
not accepted as an inline claim by the stage owner: the target checkpoint must
reference a completed `approval-decision` source checkpoint whose
`metadata.actor_role` is the approver, whose review passed, whose path and stage
match the expected approval gate, and whose approval fields match the target
artifact. The target checkpoint also requires the source checkpoint to be
registered by the runtime approval gate in the current execution, so a
hand-written matching JSON file is not accepted as approval provenance.
`human_operator` approvals are not generated by that runtime gate. They must be
provided as explicit `human_approval_grants`; without one, the stage writes an
`awaiting_human` checkpoint and stops instead of completing. Malformed grant
containers fail contract validation so configuration errors do not masquerade as
normal human approval pauses.
Checkpoint metadata records manifest-declared `tools_available` and
actual `tools_used`, and completed checkpoints reject tool usage outside the
stage manifest.
Negative replay scenarios mutate known-good outputs to prove that
contract-critical regressions are detected. Static fixtures must pass both
schema validation and semantic validation; this catches schema-valid false
greens such as commit-SHA mismatches and blocked autonomy reports
mislabeled as executed.

Real repository bug runs are represented as `real_repo_bug_run` artifacts. The
artifact separates live external proof from normal replay: the live script
captures failing and passing command evidence once, while verify and replay use
the checked-in fixture to keep daily gates deterministic.

Corpus readiness is represented separately as `real_repo_corpus_report`. This
keeps a single successful dogfood run from being misread as broad external
validity: the corpus report counts non-author repositories and failure
families, rejects duplicate artifact/run IDs, and records a `freshness` object
showing whether the current producer regenerated and semantically validated the
corpus report. The producer prefers live `.apk/real-bug-runs/<run_id>`
artifacts only when every manifest entry has one and the failing/passing command
timestamps are inside the freshness window, and marks the corpus as
`live_external_rerun`; otherwise it records stale live counts and falls back to
checked-in manifest artifacts. Each report entry carries the failing and
passing command timestamps, and semantic validation recomputes the
report-level oldest/latest command timestamps from those entries. Self-assessment
caps `evidence_trust` below perfect while live external rerun artifacts are
stale, even when the checked-in corpus still passes.
When the target is not met, the report must also carry an
approval boundary with `status=awaiting_human`, requested actions, and
operations prohibited without approval. Once the target is met, the boundary
must become `not_required`. Semantic validation rejects reports that hide
either state or claim target readiness without a passing producer roundtrip.

The current broad corpus uses five public BugsInPy-backed third-party
repositories: `youtube-dl`, `tornado`, `ansible`, `thefuck`, and `black`.
`scripts/run_bugsinpy_corpus.py` reruns those cases locally, captures command
evidence, and writes normal kernel checkpoints and `real_repo_bug_run`
artifacts.

Checkpoint status is part of the control plane. Besides `completed` and
`in_progress`, the kernel records `awaiting_human`, `failed`, and `blocked`
branches. Replay fixtures assert those branches directly so approval stops,
test failures, and external-boundary blocks are inspectable system states.

The `kernel-autonomy` pipeline closes the dogfood loop without removing human
governance. Its `autonomy_run_report` records the selected next action,
executed allowlisted commands, safety boundaries, and stop reason. The runner
can advance low-risk local verification work, but external and destructive
boundaries become blocked checkpoints.

The battle layer is a validation layer for the kernel, not an execution
platform. `battle_report` preserves the multi-judge score, dissent, and
recommended actions. `agent_battle_harness_report` makes Agent Battle
evaluation auditable with context hashes, blind-review flags, source labels,
cross-examination records, critic veto state, judge-audit checks, and an
outcome gate. A local harness derived from `self_assess` and `battle_report`
must label itself as `evidence_mode=derived_report` and cannot advance. Only
explicit external agent judge reports can set
`evidence_mode=independent_agent_reports` and `independent_contexts=true`.
Those judge reports must validate as `agent_judge_report`, bind to the same
run ID, carry advancing verdicts and scores at or above the minimum, contain
exactly one report for each required role, and use distinct
`codex-subagent://` sources. The harness also records each judge report's
`source_artifact` and `source_artifact_sha256`; semantic validation requires
that file to be readable, hash-matched, and payload-matched to the judge entry,
so an opaque source label alone cannot advance the battle. Self-assessment only accepts independent battle
evidence that references real input reports, binds the recorded
`self_assessment_run_id` and `battle_report_run_id` to those files' actual
`run_id` values, confirms all non-evaluation self-assessment dimensions meet
the pre-battle readiness gate, confirms the bound reports have no unresolved
non-battle next action, and is newer than the current source/schema/test/
documentation evidence. The bound self/battle reports also carry an
`evidence_fingerprint`; self-assessment rejects an older harness when a newer
unresolved battle-cycle report exists for that same fingerprint. The harness
audit and semantic validation both block
`advance` when those input bindings are not file-backed or below the pre-battle
readiness gate. Semantic validation rejects peer-score leakage under blind
review, duplicate judge roles, below-minimum advance scores, derived-report
`advance`, and `advance` while a veto or judge-audit failure is active.
