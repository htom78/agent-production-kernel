# Self-Driving Operating Loop

The project should practice the system it is building. A human should set
direction and approve high-risk boundaries, but routine progress should be
proposed by the kernel itself.

## Principle

The kernel is not complete when it has code. It is complete when it can inspect
its own state, score the gaps, propose the next work, execute through gates, and
turn the result into replayable evidence.

```text
self_assess -> next_actions -> implement -> verify -> replay -> battle -> schema/golden update
```

## Agent Contract

At the start of a continuation, an agent should run:

```bash
python3 scripts/self_assess.py
python3 scripts/battle_report.py
```

Then it should take the highest-priority actionable `next_actions[]` item unless
a newer human instruction overrides it. The self-assessment report is written
under:

```text
.apk/self-assessments/<run_id>/self_assessment_report.json
```

After implementation, the agent should run the full verification chain:

```bash
python3 -m unittest discover -s tests -v
python3 scripts/verify.py
python3 scripts/verify_real_repo_corpus.py
python3 scripts/replay_regressions.py
python3 scripts/battle_report.py
```

The battle report is written under:

```text
.apk/battle-reports/<run_id>/battle_report.json
```

## Scoring Dimensions

- `contract_integrity`: manifests, schemas, tools, roles, and verifier health.
- `execution_reality`: how much work is produced stage-by-stage by executors
  rather than prebuilt demo payloads.
- `evidence_trust`: whether command evidence and negative tests make false
  greens difficult.
- `replay_strength`: whether golden scenarios validate persisted checkpoint
  artifacts, contract-critical fields, and negative mutation cases.
- `multi_agent_readiness`: whether role ownership, handoffs, and approvals are
  enforced rather than only documented.
- `autonomy_loop`: whether the project can score itself and select the next
  action without waiting for a human to restate the roadmap.
- `evaluation_independence`: whether Agent Battle evidence comes from explicit
  external judge reports rather than a report derived from internal scores.

Scores are not allowed to mean “production proven” merely because fixtures are
present. Readiness is capped until the system has pack-agnostic execution,
semantic false-green tests, live producer roundtrips, and a broader real-repo
bug corpus.

The real-repo corpus gate is a readiness report rather than a local failure:
`scripts/verify_real_repo_corpus.py` must pass schema and semantic validation.
When `target_met=false`, the generated
`real_repo_corpus_report.approval_boundary` must remain `awaiting_human` and
must explicitly prohibit third-party repository selection, cloning, test runs,
credential use, paid services, and branch/pull-request operations without
approval. When the corpus reaches five non-author repositories and three
failure families, the boundary must flip to `not_required`. The report also
records `freshness`: default verification regenerates and validates the corpus
report from the current producer, preferring live artifacts under
`.apk/real-bug-runs/<run_id>/real_repo_bug_run.json` when every manifest entry
has one and the command timestamps are inside the freshness window, and
otherwise falling back to checked-in manifest artifacts with stale live counts
recorded. Each corpus entry records the failing and passing command timestamps,
and semantic validation recomputes report-level oldest/latest timestamps from
those entries. Self-assessment caps `evidence_trust` below perfect while live
external rerun artifacts are stale. Live third-party test reruns remain an
explicit external boundary.

The current external corpus is generated by
`scripts/run_bugsinpy_corpus.py --write-examples`, using public BugsInPy
metadata and isolated local venvs. The runner may clone public repositories and
run focused tests only after human approval; it must not use credentials, paid
services, pushes, branches, or pull requests.

## Governance Boundary

Self-driving does not mean unbounded autonomy. The kernel may propose and
execute low-risk improvements, but it should stop or request human approval
when a next action changes external integrations, broad project scope, cost,
credentials, or destructive runtime behavior.

The runtime may generate machine-role approvals through the approval gate, but
it must not synthesize `human_operator` approval. Stages that require human
approval need an explicit `human_approval_grants` entry; otherwise they stop as
`awaiting_human` checkpoints with the requested approver recorded in metadata.

The human remains the board. The kernel becomes the operating system.

## Battle Harness

`scripts/battle_report.py` is the score and disagreement report. It should
remain simple and readable.

`scripts/agent_battle_harness.py` is the Agent Battle validation artifact for
this project. It generates `agent_battle_harness_report`, which records:

- per-judge context hashes and evidence source labels;
- blind-review enforcement through `peer_scores_visible=false`;
- cross-examination challenges and responses;
- critic veto state;
- judge-audit checks;
- a final outcome gate.

The local harness is deterministic and useful as an auditable report, but it is
not allowed to claim independent Agent Battle evidence. When it derives judges
from `battle_report`, it must set `evidence_mode=derived_report`,
`independent_contexts=false`, and hold the outcome. To advance, the harness must
ingest explicit external judge reports, label every judge as
`external_agent_report`, and set `evidence_mode=independent_agent_reports`.
External judge inputs use the `agent_judge_report` schema before they are folded
into the final harness report. Independent mode must contain exactly one report
for each required judge, every judge must carry an advancing verdict and a score
at or above the protocol minimum, and `source_report` values must be distinct
`codex-subagent://` references.

Self-assessment treats independent battle evidence as current only when the
harness report references real self-assessment and battle-report files, was
generated after those inputs, binds `self_assessment_run_id` and
`battle_report_run_id` to those files' actual `run_id` values, confirms all
non-evaluation self-assessment dimensions meet the pre-battle readiness gate,
confirms the bound reports have no unresolved non-battle next action, and is
newer than the current source, schema, test, and documentation evidence. The
`battle_report` release-discipline action to maintain the verified corpus and
rerun gates is allowed as a pre-battle input when those gates have been rerun;
unknown next actions still block advance.
bound self/battle reports also carry an `evidence_fingerprint`, and
self-assessment rejects an older harness when a newer unresolved battle-cycle
report exists for that same fingerprint. The
harness audit and semantic validator also block `advance` when those input
bindings are not file-backed or below the pre-battle readiness gate. This lets
an independent battle resolve `evaluation_independence` without requiring that
dimension to be advanced before the battle can run. A valid old battle artifact
is useful history, but it cannot silently promote the current run.
Independent hold or veto reports remain valid evidence and should be preserved;
they move the next action from rerunning battle to addressing the recorded
findings. Only an advancing independent report can fully satisfy
`evaluation_independence`.
APK remains the production-control kernel, not the battle platform.

## Real Repo Boundary

`scripts/run_real_bug_scenario.py` is the current external proof. It is allowed
only after the human approves the public repository boundary. The resulting
`real_repo_bug_run` artifact records the public repo URL, failing branch,
passing branch, command evidence, and checkpoint refs so replay can guard the
proof without re-running network or GitHub operations during normal verify.

When a next action hits a boundary, the system should preserve it as a
checkpoint state rather than erase it from history:

- `awaiting_human`: the next step is safe only after explicit approval.
- `failed`: the step ran and produced failing evidence.
- `blocked`: credentials, network, cost, destructive behavior, or user choice
  prevents safe continuation.

The `checkpoint_branch_replay` fixture and golden scenarios make those branches
part of normal replay.

## Bounded Runner

`scripts/run_next_action.py` is intentionally conservative. It reads a
self-assessed action, checks it against a local allowlist, and writes an
`autonomy_run_report` plus a `kernel-autonomy` checkpoint.

The runner may execute local verification commands such as:

```bash
python3 scripts/verify.py
python3 scripts/replay_regressions.py
python3 -m unittest discover -s tests -v
python3 -m compileall apkernel scripts tests
```

It must stop with `decision=blocked` when the selected action involves a real
external repository, credentials, network cost, destructive operations, unknown
handlers, or a user choice.
